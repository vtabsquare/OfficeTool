2.1 MAIN SERVER: unified_server.py
-----------------------------------
Size: ~15,000 lines
Purpose: Central API server handling all HTTP requests

Key Imports:

- Various blueprints for modular routes

Global Configuration:
- RESOURCE: Dataverse API endpoint

Performance Middleware (Lines 55-69):
- Tracks request timing
- Logs slow requests (>500ms)
- Helps identify performance bottlenecks

Authentication Functions:
- admin_required(): Decorator for admin-only routes
- Validates JWT token and checks designation for admin/manager role

/api/admin/query (POST)
- Purpose: Admin-only direct Dataverse query endpoint
- Flow: Validate admin → Build OData query → Execute → Return results
- Security: Requires admin designation

/api/employee/list (GET)
- Purpose: List all employees with pagination
- Flow: Get token → Query Dataverse → Format response
- Returns: Employee list with metadata

/api/employee/<employee_id> (GET)
- Purpose: Get single employee details
- Flow: Validate token → Fetch by ID → Return formatted data

/api/employee/create (POST)
- Purpose: Create new employee record
- Flow: Validate data → Generate employee ID → Create in Dataverse
- Auto-generates: Employee ID, user credentials

/api/employee/update (POST)
- Purpose: Update employee details

5. USER FLOW ANALYSIS
================================================================================

5.1 LOGIN FLOW:
--------------
1. User enters credentials on login page
2. Frontend sends POST to /api/login
3. Backend validates against Dataverse
4. JWT token generated and returned
5. Frontend stores token and redirects to dashboard
6. All subsequent requests include Bearer token

5.2 DASHBOARD FLOW:
------------------
1. After login, user lands on dashboard
2. Multiple API calls in parallel:
   - /api/attendance/status - Current attendance
   - /api/leave/balance - Leave balance
   - /api/time-tracker/status - Active timer
   - /api/chat/unread - Unread messages
   - /api/tasks/my-tasks - Assigned tasks
3. Widgets display real-time data
4. Auto-refresh every 30 seconds

5.3 ATTENDANCE CHECK-IN FLOW:
----------------------------
1. User clicks "Check In" button
2. Browser requests geolocation
3. Frontend sends POST to /api/attendance/checkin
   with location and timestamp
4. Backend validates:
   - No active check-in exists
   - Within working hours (optional)
   - Location within allowed radius
5. Record created in Dataverse
6. UI updates to show "Check Out" button
7. Timer starts showing duration

5.4 ATTENDANCE CHECK-OUT FLOW:
-----------------------------
1. User clicks "Check Out" button
2. Frontend sends POST to /api/attendance/checkout
3. Backend:
   - Finds active check-in record
   - Calculates total duration
   - Updates record with checkout time
   - Handles overtime calculation
4. UI shows today's summary
5. Timer stops

5.5 LEAVE APPLICATION FLOW:
---------------------------
1. User navigates to Leave module
2. Clicks "Apply for Leave"
3. Form appears with fields:
   - Leave type (dropdown)
   - Start date (date picker)
   - End date (date picker)
   - Reason (textarea)
4. Frontend validates:
   - Dates are valid
   - Not past dates
   - Within allowed limits
5. On submit, POST to /api/leave/request
6. Backend checks:
   - Leave balance sufficiency
   - No overlapping requests
   - Minimum notice period
7. Request created with "pending" status
8. Email sent to manager
9. UI shows "Pending Approval"

5.6 LEAVE APPROVAL FLOW (MANAGER):
----------------------------------
1. Manager receives email/notification
2. Navigates to Leave → Team Leaves
3. Sees pending requests with details
4. Clicks "Approve" or "Reject"
5. If rejecting, must provide reason
6. POST to /api/leave/approve
7. Backend:
   - Validates approver authority
   - Updates leave status
   - Adjusts leave balance if approved
   - Sends notification to employee
8. UI updates in real-time for employee

5.7 TIME TRACKING START TIMER FLOW:
----------------------------------
1. User goes to Time Tracking module
2. Selects task from dropdown
3. Clicks "Start Timer"
4. Frontend sends POST to /api/time-tracker/start
5. Backend:
   - Stops any other active timers
   - Creates new time entry
   - Returns entry ID
6. UI shows running timer
7. Updates every second

5.8 TIME TRACKING STOP TIMER FLOW:
---------------------------------
1. User clicks "Stop Timer"
2. Frontend sends POST to /api/time-tracker/stop
3. Backend:
   - Finds active timer
   - Calculates duration
   - Saves to timesheet
   - Updates task status if needed
4. Entry appears in timesheet list
5. Timer resets

5.9 PROJECT CREATION FLOW:
-------------------------
1. Admin/Manager clicks "New Project"
2. Modal opens with form fields:
   - Project name
   - Description
   - Start date
   - End date
   - Team members
3. On submit, POST to /api/projects/create
4. Backend:
   - Validates permissions
   - Generates project ID
   - Creates project record
   - Creates default Kanban board
   - Adds team members
5. Redirects to project details

5.10 TASK CREATION FLOW:
------------------------
1. User in project view clicks "Add Task"
2. Form appears with:
   - Task title
   - Description
   - Assignee
   - Priority
   - Due date
3. POST to /api/tasks/create
4. Backend:
   - Generates task ID
   - Creates task record
   - Adds to "To Do" column
5. Task appears on Kanban board

5.11 CHAT FLOW:
--------------
1. User clicks chat icon
2. Chat window opens
3. Selects contact or room
4. Types message and hits Enter
5. WebSocket emits message
6. Backend:
   - Saves to database
   - Broadcasts to recipients
7. Messages appear in real-time

5.12 MEETING SCHEDULING FLOW:
----------------------------
1. User clicks "Schedule Meeting"
2. Form with:
   - Title
   - Participants
   - Date/Time
   - Duration
3. On submit, POST to /api/meet/create
4. Backend:
   - Creates Google Calendar event
   - Generates meeting link
   - Sends invitations
5. Meeting appears in calendar

7. KEY BUSINESS LOGIC
================================================================================

7.1 ATTENDANCE CALCULATIONS:
---------------------------

Working Hours Definition:
- Standard: 9 hours (9:00 AM - 6:00 PM)
- Grace period: 15 minutes
- Overtime: After 9 hours

Late Mark Rules:
- After 9:15 AM = Late
- After 10:00 AM = Half-day
- After 11:00 AM = Absent

Early Exit Rules:
- Before 6:00 PM = Early exit
- Before 3:00 PM = Half-day

Auto-Checkout Logic:
- Runs at midnight
- Closes open sessions
- Assumes checkout at 6:00 PM
- Flags for manual review

7.2 LEAVE CALCULATIONS:
-----------------------

Leave Accrual Rules:
- CL (Casual Leave): 6 days/year
- SL (Sick Leave): 6 days/year
- PL (Privileged Leave): Based on experience
  - < 1 year: 0 days
  - 1-2 years: 12 days
  - 2-5 years: 18 days
  - > 5 years: 24 days

Leave Encashment:
- Only PL can be encashed
- Maximum 10 days per year
- Requires 1 year service

Leave Carry Forward:
- CL/SL: Not carried forward
- PL: Max 15 days carried forward

7.3 TIMESHEET RULES:
-------------------

Minimum Entry: 15 minutes
Maximum Daily: 12 hours
Weekend Rates: 1.5x
Holiday Rates: 2x

Approval Workflow:
- Self-submit → Manager review → Approved
- Auto-approve for < 8 hours
- Reject with reason required

7.4 PROJECT MANAGEMENT:
-----------------------

Project Status Flow:
- Planning → In Progress → On Hold → Completed → Closed

Task Priority Levels:
- Critical (P0) - Immediate attention
- High (P1) - This week
- Medium (P2) - This month
- Low (P3) - When possible

Task Status Flow:
- To Do → In Progress → Review → Done → Archived

7.5 PERFORMANCE METRICS:
------------------------

Attendance Score:
- Present: 100%
- Late: 75%
- Half-day: 50%
- Absent: 0%

Productivity Index:
- Based on task completion
- Quality rating
- Deadline adherence
