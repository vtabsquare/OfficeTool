HR TOOL END-TO-END ANALYSIS AND MYSQL MIGRATION PLANNER
=====================================================

Project analyzed: OfficeTool
Current hosting guidance found in repo: DigitalOcean App Platform deployment guide
Current live data architecture: Primarily Microsoft Dataverse via REST/OData, plus some local file-based storage, plus one small SQLAlchemy token store.

====================================================================
1. EXECUTIVE SUMMARY
====================================================================

This application is not currently a normal Flask app backed by one relational database.
It is a hybrid system with these active storage patterns:

1) Main business data is stored in Microsoft Dataverse
2) Some auxiliary/workflow data is stored in local JSON/CSV files in backend/_data and backend/storage
3) Google OAuth token persistence uses SQLAlchemy and currently defaults to SQLite unless DATABASE_URL is provided
4) The app is deployed as a multi-service web app architecture:
   - Frontend: Vite static site
   - Backend: Flask API
   - Socket server: Node.js / Socket.IO

Because of this, migrating “the DB” to MySQL is not a single-step database export/import task.
It is a platform migration and application refactor project.

The biggest reality to understand:
- The application logic is tightly coupled to Dataverse table names, Dataverse field names, Dataverse record GUIDs, and OData query behavior.
- To move to MySQL, you must migrate both:
  a) the data
  b) the application code that reads/writes that data

Recommended approach:
- Do NOT perform a big-bang production switch directly from Dataverse to MySQL.
- Build a compatibility data-access layer first.
- Migrate module by module.
- Run dual-read/dual-write or staged cutover where practical.
- Keep rollback ready until the new MySQL path is validated in production.

4. CURRENT BUSINESS MODULES AND THEIR DATA LOGIC
====================================================================

4.1 Employee Management
-----------------------
Observed in unified_server.py.
Employee entity is auto-resolved between possible Dataverse tables such as:
- crc6f_table12s
- crc6f_employees
- related alternatives

Employee-related logic includes:
- employee listing
- employee add/update/delete/restore flows
- role/designation usage for admin checks
- photo/profile handling
- active/inactive flag logic
- deleted employee archival in CSV
- leave balance / leave quota relationships
- team hierarchy/cache interactions

Migration impact:
- Employee table will become one of the anchor MySQL tables.
- All field mapping assumptions in code must be normalized.
- Deleted_employees.csv should become a proper archive or soft-delete strategy in MySQL.

4.2 Attendance
--------------
Main live attendance logic appears in:
- backend/attendance_service_v2.py
- backend/attendance_scheduler.py

Current design:
- Backend-authoritative attendance
- Dataverse attendance table used
- Dataverse login activity table used
- Server-side check-in / check-out calculation
- Auto-close stale sessions at midnight
- Absent marking job
- Attendance events emitted to socket server
- Attendance logic also force-stops active task timers

Dataverse entities seen:
- crc6f_table13s (attendance)
- crc6f_hr_loginactivitytbs (session tracking)
- employee master table references

Key business rules identified:
- Attendance status derived from total seconds worked
- >= 9 hours => Present
- >= 4 hours => Half Day
- otherwise Absent
- Timezone-aware date determination
- Auto-close older open sessions at local midnight
- Scheduler marks absent employees if no attendance exists

Migration impact:
- Attendance is one of the highest-risk modules.
- It contains date/time, timezone, scheduling, session, and reconciliation logic.
- Must be one of the first properly modeled MySQL modules.

4.4 Projects / Boards / Columns / Tasks
---------------------------------------
Modules:
- backend/project_boards.py
- backend/project_column.py
- backend/project_tasks.py
- backend/project_contributors.py

Current design:
- Projects stored in Dataverse project headers
- Boards stored in a Dataverse board table
- Columns/statuses stored in another Dataverse table
- Tasks stored in Dataverse task details table
- Contributors stored in Dataverse contributor table
- Some counts are denormalized or recomputed dynamically
- Board delete cascades manually to tasks
- Column rename updates tasks with old status to new status
- Contributor recount updates project header contributor count
- IDs like BRD001, TASK001, BUG001, REC001 are generated in code

Migration impact:
- This area maps well to relational tables.
- Manual cascade behavior can be formalized with foreign keys and transaction boundaries.
- Generated business IDs need unique constraints and sequence-safe generation.
- Existing duplicate checks should move to DB constraints where possible.

4.5 Time Tracking / Timesheets

Migration impact:

6. DOES DIGITALOCEAN PROVIDE MYSQL CLOUD DATABASE?
====================================================================

Yes. DigitalOcean does provide managed cloud databases, including MySQL.

You generally have two options on DigitalOcean:

OPTION A: Managed Database (Recommended)
- DigitalOcean Managed MySQL
- Automatic backups
- Automated patching
- Connection pooling options depending on plan/features
- High availability options depending on plan/tier
- SSL/TLS support
- Better for production than self-managed MySQL on a droplet

OPTION B: Self-managed MySQL on a Droplet
- Cheaper at low scale
- More operational burden
- You handle backups, upgrades, failover, security, tuning
- Not recommended for critical production HR data unless you have ops maturity

Recommendation for this app:
- Use DigitalOcean Managed MySQL for production.
- Keep app services on App Platform or Droplets, but store business data in managed MySQL.

7. HIGH-LEVEL MYSQL TARGET ARCHITECTURE
====================================================================

Recommended target:

1) Frontend
   - Keep as is

2) Backend Flask API
   - Refactor to use SQLAlchemy or a well-structured repository/data-access layer
   - Stop calling Dataverse directly for migrated modules

3) Socket server
   - Keep as is initially
   - Continue emitting real-time events

4) Managed MySQL
   - Single source of truth for operational HR data

5) Object/File storage (optional but recommended)
   - Use DigitalOcean Spaces or another object store for large files/attachments/documents
   - Store metadata and object keys in MySQL

6) Redis (optional)
   - Good for ephemeral sessions, queues, caching, timers, locks
   - Useful later, not mandatory for first migration

8. RECOMMENDED MIGRATION STRATEGY
====================================================================

Do this in phases, not in one production switch.

PHASE 0 - DISCOVERY AND FREEZE
------------------------------
Goal:
- Fully inventory data model, production traffic, and active modules.

Tasks:
- Identify exact live backend entrypoint in production
- Confirm which modules are actually used
- Export a complete list of Dataverse entities in use
- Export record counts for each entity
- Identify all local file stores used in production
- Identify all scheduled/background jobs
- Identify all external integrations:
  - Azure AD / MSAL / Dataverse
  - Google Calendar OAuth
  - SMTP mail
  - Socket server
- Freeze schema-changing feature work during migration planning window

Deliverables:
- Source-to-target entity mapping spreadsheet
- Production data inventory
- Critical-path module list

PHASE 1 - DESIGN TARGET MYSQL SCHEMA
------------------------------------
Goal:
- Design normalized MySQL schema that supports the current business logic.

Suggested table groups:
- employees
- employee_documents or employee_files
- deleted_employee_archive or employee_status_history
- leave_requests
- leave_balances
- leave_types
- comp_off_requests
- holidays
- attendance_days
- attendance_sessions
- projects
- project_boards
- project_columns
- project_tasks
- project_contributors
- time_entries
- timesheet_logs
- timesheet_submissions
- timesheet_approvals
- conversations
- conversation_members
- messages
- message_statuses
- attachments
- assets
- team_hierarchy
- google_tokens
- audit_logs

Important design principles:
- Keep internal numeric or UUID primary keys
- Preserve business IDs like EMP001, TASK001, BRD001, REC001 with unique constraints
- Add created_at and updated_at consistently
- Add indexes for commonly queried fields
- Add foreign keys where relationships are certain
- Add status history/audit tables for sensitive HR flows where needed
- Avoid storing derived counts unless required for performance

PHASE 2 - BUILD DATA ACCESS ABSTRACTION IN CODE
-----------------------------------------------
Goal:
- Stop business logic from depending directly on Dataverse/OData.

This is the most important engineering step.

What to do:
- Introduce a repository/service layer, for example:
  - EmployeeRepository
  - AttendanceRepository
  - LeaveRepository
  - ProjectRepository
  - TimesheetRepository
  - ChatRepository
  - AssetRepository

Then business code calls repository methods instead of Dataverse directly.

Example direction:
- Existing code: requests.get(RESOURCE + '/api/data/v9.2/...')
- New code: employee_repo.get_by_employee_id(...)

Why this matters:
- Without this layer, migration will become brittle and error-prone.
- With this layer, you can switch implementations per module:
  - Dataverse-backed implementation
  - MySQL-backed implementation

Recommended transitional approach:
- Keep current endpoints unchanged for frontend compatibility
- Only change persistence internals first

PHASE 3 - STAND UP DIGITALOCEAN MANAGED MYSQL
---------------------------------------------
Goal:
- Create production-grade MySQL environment.

Steps in DigitalOcean:
1) Log in to DigitalOcean Cloud dashboard
2) Go to Databases
3) Create Database Cluster
4) Select MySQL
5) Choose production-appropriate version
6) Choose region close to app servers
7) Choose node size based on current dataset and traffic
8) Enable trusted sources / networking rules
9) Create database name, user, password
10) Download SSL CA cert if required by client setup
11) Note host, port, database name, username, password

Recommended environment variables:
- MYSQL_HOST
- MYSQL_PORT
- MYSQL_DATABASE
- MYSQL_USER
- MYSQL_PASSWORD
- MYSQL_SSL_CA
- DATABASE_URL

Suggested SQLAlchemy URL pattern:
- mysql+pymysql://USER:PASSWORD@HOST:PORT/DBNAME
or
- mysql+mysqlconnector://USER:PASSWORD@HOST:PORT/DBNAME

Recommended operational features:
- Daily backups enabled
- Point-in-time recovery if available in your chosen plan
- Connection limits reviewed
- Alerting enabled
- Staging DB cluster separate from production DB cluster

PHASE 4 - IMPLEMENT MYSQL SCHEMA + MIGRATIONS
---------------------------------------------
Goal:
- Create schema in a controlled, repeatable way.

Recommended toolset:
- SQLAlchemy ORM for models
- Alembic for schema migrations

Why:
- You need versioned schema changes for production reliability
- Manual SQL files alone become risky over time

Tasks:
- Create models
- Create initial Alembic migration
- Add indexes and constraints
- Validate schema in staging
- Create seed/reference data migration where needed

PHASE 5 - DATA EXTRACTION FROM DATAVERSE AND FILES
--------------------------------------------------
Goal:
- Extract current production data into portable form.

For Dataverse data:
- Export entity by entity using scripts or Dataverse APIs
- Preserve:
  - Dataverse GUIDs where useful for traceability
  - business IDs
  - timestamps
  - status values
  - foreign relationships

For local files:
- Export JSON/CSV into structured intermediate datasets
- Clean duplicates and malformed records before MySQL import

Recommended approach:
- Create one export script per domain:
  - export_employees.py
  - export_attendance.py
  - export_leaves.py
  - export_projects.py
  - export_timesheets.py
  - export_chat.py
  - export_assets.py
  - export_local_files.py

Store exports as:
- CSV for simple tables
- JSON for hierarchical records

Important:
- Do not transform too aggressively during extraction.
- Keep a raw export snapshot for rollback and auditing.

PHASE 6 - TRANSFORM + LOAD INTO MYSQL
-------------------------------------
Goal:
- Convert source formats into the new target schema.

Tasks:
- Map Dataverse field names to clean SQL column names
- Normalize mixed date/time formats
- Convert string numbers to numeric columns
- Standardize status enums
- Resolve duplicate IDs
- Resolve missing foreign keys
- Handle orphaned records
- Import in dependency order

Recommended load order:
1) employees
2) leave types / holidays / reference tables
3) projects
4) boards
5) columns
6) contributors
7) tasks
8) assets
9) attendance days
10) attendance sessions/login activity
11) time entries / timesheet logs / submissions
12) chat conversations/members/messages
13) google tokens
14) hierarchy / admin metadata / archives

Important validation after each load:
- source count vs target count
- random sample comparison
- business-key uniqueness checks
- foreign key integrity checks

PHASE 7 - STAGING CUTOVER TEST
------------------------------
Goal:
- Prove the full app works on MySQL in a staging environment.

Set up staging with:
- cloned backend
- cloned socket server
- staging frontend if needed
- staging managed MySQL
- production-like env vars
- sanitized production snapshot if allowed

Then test:
- login
- employee list/add/edit/delete/restore
- attendance checkin/checkout/status/monthly history
- midnight auto-close scheduler
- absent marking job
- leave apply/cancel/approve/reject
- project create/update/delete
- board/column/task CRUD
- contributor add/remove
- task timer and timesheet workflows
- asset CRUD
- chat send/read/reply/search
- Google calendar token persistence
- password reset and email sending
- notifications and socket events

PHASE 8 - PRODUCTION DEPLOYMENT STRATEGY
----------------------------------------
Goal:
- Migrate live production with minimum downtime and rollback safety.

Recommended production strategy:

OPTION 1: Module-by-module cutover
- Migrate least risky modules first
- Example order:
  1) Google tokens
  2) leave types / holidays / assets
  3) projects/tasks
  4) employee master
  5) leave management
  6) attendance
  7) timesheets
  8) chat
- Safest from application perspective
- Takes longer

OPTION 2: Big coordinated cutover
- Export final Dataverse snapshot
- Freeze writes briefly
- Load MySQL
- Flip app to MySQL backend
- Validate immediately
- Only suitable after excellent staging confidence

For this HR app, recommended choice:
- Module-by-module or domain-by-domain staged cutover
- Attendance, leave, and employee data are too critical for a reckless big-bang switch

PHASE 9 - DUAL-WRITE OR SHADOW MODE (OPTIONAL BUT IDEAL)
--------------------------------------------------------
Goal:
- Reduce risk by comparing Dataverse and MySQL behavior before full cutover.

Approach:
- Reads still from Dataverse
- Writes go to Dataverse and MySQL
- Compare results in logs/reconciliation jobs

Or:
- Certain modules read from MySQL while fallback remains Dataverse

Benefits:
- Early confidence
- Easier rollback
- Easier data comparison

Trade-off:
- More engineering complexity

PHASE 10 - FINAL CUTOVER
------------------------
Goal:
- Make MySQL the source of truth.

Tasks:
- Schedule maintenance/change window
- Take final Dataverse export backup
- Freeze writes briefly if needed
- Run final incremental sync
- Switch env vars and backend repository implementation
- Deploy backend
- Smoke test critical flows immediately
- Monitor logs, errors, queue/socket behavior, DB load

PHASE 11 - POST-CUTOVER STABILIZATION
-------------------------------------
Goal:
- Ensure no silent corruption or workflow failures.

Tasks:
- Daily reconciliation reports for 1-2 weeks
- Monitor attendance totals vs expected
- Monitor leave requests and approvals
- Monitor timesheet submissions and approvals
- Monitor chat delivery and persistence
- Monitor scheduler runs and absent marking
- Monitor DB performance and slow queries

PHASE 12 - DECOMMISSION DATAVERSE DEPENDENCIES
----------------------------------------------
Goal:
- Remove technical debt safely.

Tasks:
- Remove unused Dataverse helper calls
- Remove Azure app dependency for business data if no longer needed
- Keep only what is needed for identity if still applicable
- Archive export scripts and migration logs
- Document final architecture

12.1 Introduce a Persistence Layer
----------------------------------
Create a structure such as:
- backend/repositories/
- backend/models/

Examples:

15. ROLLBACK PLAN
====================================================================

You must define rollback before production cutover.

Rollback essentials:
- Keep Dataverse unchanged until confidence is high
- Keep export snapshot before every major import
- Keep feature flags to switch back to Dataverse quickly
- Do not delete Dataverse data during initial cutover period
- Keep DB backups before schema changes and before production import

Rollback scenarios:
- If MySQL write path fails for a module, switch that module back to Dataverse
- If data integrity issue found, restore from backup and replay from validated snapshot
- If performance issue found, revert module read path to Dataverse while tuning MySQL

16. RISK REGISTER
====================================================================

Risk 1: Dataverse field semantics do not match clean relational assumptions
Mitigation:
- Build mapping doc
- validate real sample records

Risk 2: Hidden production data in local JSON/CSV files gets forgotten
Mitigation:
- inventory every file-backed store
- inspect production filesystem contents before cutover

Risk 3: Duplicate / legacy backend modules cause migration of wrong code path
Mitigation:
- confirm live entrypoint and used endpoints from deployment/start command and logs

Risk 4: Attendance timezone/session logic breaks after migration
Mitigation:
- write automated tests using real edge cases around midnight and timezone transitions

Risk 5: Business IDs collide in MySQL sequences
Mitigation:
- add unique constraints
- centralize ID generation
- backfill next-sequence values after import

Risk 6: Chat attachments become oversized for DB
Mitigation:
- use object storage for files, keep metadata in DB

Risk 7: Long-running migration causes drift between Dataverse and MySQL
Mitigation:
- use delta sync and/or dual-write during transition

Risk 8: Scheduler behavior changes after migration
Mitigation:
- test midnight auto-close and absent marking in staging with seeded data
